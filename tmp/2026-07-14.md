‚óŹ Confirmed the root cause: _latex_to_omath extracts <m:oMath> from a
  pandoc-generated WordprocessingML document, and because that element is still
  attached (in libxml2's tree) to the w:document root which declares xmlns:w,
  xmlns:o, xmlns:v, xmlns:w10, xmlns:pic, xmlns:wp, serializing it via
  etree.tostring() drags all those inherited Word-specific namespace
  declarations along ‚ÄĒ even though none are actually used inside the subtree.
  The result is <m:oMath xmlns:w="..." xmlns:o="..." xmlns:v="..." 
  xmlns:w10="..." xmlns:pic="..." xmlns:wp="..."> embedded inside a
  PresentationML part. Real PowerPoint's stricter OOXML/MC parser chokes on
  foreign WordprocessingML namespace declarations inside the a14:m/mc:Choice
  math branch and blanks the shape, while LibreOffice's lenient parser (which
  never even evaluates that branch, since it doesn't support Requires="a14" and
  always renders mc:Fallback) never exposed the problem. The stale sidebar
  thumbnail is a cached render from a point where the slide XML was still being
  parsed for text extraction, not evidence the math itself renders live.

  The fix is to strip unused namespace declarations from the extracted m:oMath
  before serializing it.
