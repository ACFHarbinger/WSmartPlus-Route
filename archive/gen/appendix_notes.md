# Route Constructors

## BPC (Branch-Price-and-Cut)

**Code**: `logic/src/policies/route_construction/exact_and_decomposition_solvers/branch_and_price_and_cut/{policy_bpc.py,bpc_engine.py,params.py}`, with heavy supporting logic in `logic/src/policies/helpers/solvers_and_matheuristics/{master_problem/model.py, pricing/solver.py, search/cutting_planes.py, search/column_generation.py, branching/strategies.py}`.

**Master formulation** (`VRPPMasterProblem.build_model`, `master_problem/model.py:210-267`): a route-column set-partitioning/set-covering LP. Binary variables `route_k` (one per generated `Route`), objective coefficient = `route.profit` (revenue minus travel cost of that route), maximized. Mandatory nodes get `Σ_k δ_ik λ_k == 1` if `strict_set_partitioning` (forced when Ryan-Foster branching is active), else `>= 1`; optional nodes get packing constraints `Σ_k δ_ik λ_k <= 1` (model.py:245-260). A fleet-size cut `Σ λ_k <= vehicle_limit` is added when a vehicle cap is supplied. No Big-M artificial variables are used — infeasible RMPs are handled via Farkas dual-ray extraction (`_handle_infeasibility`) to drive Phase-I pricing (Barnhart et al. 2000 §3), and the code explicitly documents that Big-M would "suppress infeasibility and deadlock the Phase I/Farkas loop." Gurobi is configured with `Method=1` (dual simplex, for LP reoptimization after adding columns/cuts) and `InfUnbdInfo=1` for ray extraction.

**Pricing subproblem**: `RCSPPSolver._label_correcting_algorithm` (`pricing/solver.py:599`) — a resource-constrained shortest-path label-correcting DP over the full node graph (depot + n customers), with load as the resource dimension, run as a max-heap on reduced cost. It supports **ng-route relaxation** (Baldacci et al. 2011) via `ng_neighborhoods` (`ng_neighborhood_size`, default 8) to keep near-elementarity without full elementary-path complexity, plus optional **DSSR** (Decremental State-Space Relaxation, Righini & Salani 2008, `enable_dssr`/`dssr_max_iters`) to iteratively add cycling vertices to ng-memory until routes become elementary. Reduced-cost formula per the module docstring: `c'_k = c_k - Σ π_i δ_ik - Σ π_S a_Sk - Σ γ_lm α_lm^k`, i.e. node duals, generic cut duals, and LCI arc-lifting duals `γ_lm` all modify arc costs inside `_extend_label` (`pricing/solver.py:898`). "Reduced-cost arc fixing" (Irnich et al. 2010, `enable_reduced_cost_arc_fixing`) prunes pricing arcs whose minimum achievable reduced cost cannot beat the current incumbent gap.

**Cuts**: `params.cutting_planes` defaults to `"saturated_arc_lci"` — deliberately not Rounded Capacity Cuts, per an explicit "Bug #1 fix" comment in `params.py:73-77` stating RCC (Lysgaard 2004) postdates Barnhart-Hane-Vance (2000) and that Lifted Cover Inequalities on saturated arcs (`SaturatedArcLCIEngine`, `cutting_planes.py:1056`) is what replicates the paper's §6 experimental setup (Tables 3-4). `PhysicalCapacityLCIEngine` (line 894) is the alternate LCI family for node/vehicle-capacity knapsacks, lifted via the Gu-Nemhauser-Savelsbergh (1995a) exact sequence-independent procedure. Beyond the paper, a `CompositeCuttingPlaneEngine` always layers in VRPP-specific supplemental engines regardless of `cutting_planes_name`: `MinCutInequalityEngine`, `TriangleCliqueCutEngine`, `LimitedMemoryRank1CutEngine` (3-SRI-style rank-1 cuts, Jepsen et al. 2008), `NodeProfitBoundEngine`, `PathEliminationEngine`. `RoundedCapacityCutEngine` (RCC), `SubsetRowCutEngine`, `EdgeCliqueCutEngine`, `RoundedMultistarCutEngine` also exist as selectable/composited engines. Cuts are filtered for near-duplication via `cut_orthogonality_threshold` (cosine-similarity ceiling, default 0.8).

**Branching**: default `branching_strategy="divergence"` = `MultiEdgePartitionBranching.find_divergence_node` (`branching/strategies.py:191`), an explicit re-implementation of BHV2000 §4.3: trace the two highest-λ fractional routes step-by-step from the depot to their first divergence node `d`; partition outgoing arcs there into `arc_set_1`/`arc_set_2` by aggregate λ-weight (a balanced-median split, going beyond the paper's simple two-arc split to avoid "thin" branches). DFS node ordering follows the paper's §3.2 rule literally (`prefer_shorter_path_dfs=True`): the child forbidding the longer path's arc set is explored first. **Before** arc/RF branching, the engine runs a VRPP-specific hierarchical **node-visitation (y_v) branching** step (`enable_node_visitation_branching=True`, citing Boussier/Feillet/Gendreau 2007 and Pessoa et al. 2020) via `HierarchicalStrongBranching.find_best_y_branching` — this is the mechanism that resolves the "should node i even be visited" question that classical BHV2000 (an all-covering flow problem) never had to answer (see VRPP adaptation below). Ryan-Foster (1981) node-pair branching (`RyanFosterBranching`) is also available and, when selected, forces `master.strict_set_partitioning = True` for polyhedral correctness. Falls back to `EdgeBranching.find_branching_arc` if divergence finds no candidate at a fractional node.

**VRPP adaptation (profitable subset selection)**: unlike BHV2000's all-node-covering ODIMCF, WSmart-Route's VRPP allows not visiting optional nodes. This is handled at three levels: (1) master problem uses `<= 1` packing (not `== 1`) for optional nodes so a column simply omitting a node is legal; (2) pricing maximizes reduced-cost profit rather than solving a pure shortest path, so RCSPP naturally drops unprofitable nodes; (3) the added **node-visitation branching** layer explicitly branches on `y_v ∈ {0,1}` (visit/don't-visit) ahead of arc-level decisions to resolve "profit-selection × routing duality." Additionally, a pre-BPC MIP knapsack heuristic, `_select_nodes_knapsack` (`bpc_engine.py:181`), can prune the optional-node set before RCSPP ever runs, using a Clarke-Wright-style insertion-cost estimate and a capacity/size-bounded 0-1 knapsack solved via Gurobi (`params.knapsack_proc_selection`); it is a heuristic size-reduction step, not exact, with a fallback to top-k by net value on solver failure.

**Other mechanisms**: Lagrangian-relaxation pre-pruning (`lr_pre_pruning`, subgradient over the uncapacitated OP relaxation, `lr_max_subgradient_iters`, `lr_op_time_limit`) prunes B&B nodes before invoking column generation; optional LR warm-start seeds CG with the λ*-optimal OP route. `cg_at_root_only` replicates BHV2000 Table 2 (CG restricted to root, later nodes only re-solve LP over the frozen pool). Column-pool deduplication and dual smoothing (Wentges 1997) are available.

**Limitations / scalability caveats visible in code**: `max_bb_nodes` default 1000, `time_limit` default 60s wall-clock (checked at both global and per-node granularity, with root node capped at 40% of remaining budget); `rcspp_timeout=30s` and `rcspp_max_labels=1e6` are hard safety caps on the pricer to prevent label-space explosion — DP is not exhaustive by default (ng-relaxation trades bound tightness for tractability). On timeout with no integer incumbent, `run_bpc` falls back to returning the mandatory-only greedy-insertion routes computed as CG warm-start columns (`bpc_engine.py:970-975`), i.e. a non-exact fallback profit, not a proven-optimal or even feasible-improved answer. `strong_branching_heuristic` is explicitly disabled by default with a code comment noting it is "a heuristic proxy that returns the top pre-sorted divergence candidate without solving any child LP relaxations" — not true strong branching — and should not be enabled without further implementation work.

**Citations found in code**: Barnhart, Hane & Vance (2000, OR 48(2):318-326) — primary architectural reference; Barnhart et al. (1998, OR 46(3):316-329); Baldacci et al. (2011, ng-route relaxation); Ryan & Foster (1981); Lysgaard et al. (2004, RCC); Jepsen et al. (2008, SRI); Gu, Nemhauser & Savelsbergh (1995a, LCI lifting); Wentges (1997, dual smoothing); Righini & Salani (2008, DSSR); Irnich et al. (2010, reduced-cost arc fixing); Boussier, Feillet & Gendreau (2007) and Pessoa et al. (2020) for node-visitation branching.

## SWC-TCF (Smart Waste Collection — Two-Commodity Flow)

**Code**: `logic/src/policies/route_construction/exact_and_decomposition_solvers/smart_waste_collection_two_commodity_flow/{policy_swc_tcf.py, dispatcher.py, gurobi.py, ortools_wrapper.py, pyomo_wrapper.py, params.py}`. Directly cites Ramos, Morais & Barbosa-Póvoa (2018, Expert Systems with Applications 103:146-158), and the code implements their "smart collection approach" model (equations 11-21), itself an extension of Baldacci, Hadjiconstantinou & Mingozzi (2004)'s two-commodity-flow CVRP formulation.

**Formulation** (`gurobi.py:_run_gurobi_optimizer`, lines 79-168; replicated in `ortools_wrapper.py` and `pyomo_wrapper.py`): binary `x[i,j]` (arc used), binary `g[i]` (node i visited — the profit/service-level selection variable), integer `k_var` (vehicles used), plus two **continuous flow commodities** on every feasible arc: `f[i,j]` (waste-load flow) and `h[i,j]` (empty-capacity flow). Capacity is enforced by `f[i,j] + h[i,j] == Q * x[i,j]` for every arc — this single constraint self-enforces both vehicle capacity and subtour elimination without separate DFJ/MTZ constraints, exactly as in Baldacci et al. (2004)'s "real depot / copy depot" trick. Flow-balance constraints for `f` (waste accumulates along the route: outflow - inflow = `S_i * g_i`) and for `h` (empty space depletes: inflow - outflow = `S_i * g_i`) are added per real node; depot balance closes the loop via the copy-depot convention.

**Objective**: standard mode maximizes `R * Σ S_i*g_i - 0.5*C*Σ x_ij*d_ij - Omega*k_var` — net revenue from collected waste minus travel cost (the `0.5` corrects for the two-commodity formulation double-counting each physical edge as two directed flow legs) minus a fixed penalty per vehicle used. A **pricing/dual mode** exists (`dual_values` argument): when duals are supplied, the objective becomes reduced-cost maximization — suggesting SWC-TCF is also usable as a pricing oracle, though nothing shows it currently wired into BPC's column generation; it appears to be a standalone capability.

**Service-level / mandatory-visit constraints** (matching paper eqs. 16-17): a soft aggregate constraint `Σ g_i (mandatory) >= |mandatory| - |nodes| * delta` bounds how many "critical" bins can be skipped, parameterized by `delta` (allowed overflow fraction); any node that is flagged mandatory or whose fill exceeds `psi * 100` gets `g_i == 1` hard-fixed. `Omega`, `delta`, `psi` map 1:1 to the paper's Ω (vehicle-use penalty), δ (service level) and ψ (overflow threshold) — this is the paper's built-in mechanism for VRPP's "not visiting a node" behavior: `g_i` is a first-class decision variable rather than an assumed-covered index, so the VRPP-with-profits behavior comes directly from the source paper rather than being a WSmart-Route bolt-on (contrast with BPC, which had to retrofit y-branching onto a covering formulation that originally assumed full coverage).

**Solver backends** (`dispatcher.py:run_swc_tcf_optimizer`): three interchangeable frameworks — `framework="gurobi"/"native"`, `framework="ortools"` (GUROBI/SCIP/HIGHS/CPLEX backends), `framework="pyomo"` (gurobi/scip/appsi_highs). All three build the identical MILP independently (constraints duplicated per-backend, not shared via a common builder — a potential drift risk). Solver-specific tuning found only in the native Gurobi path: `MIPFocus=1`, `Heuristics`, `Cuts=3`, explicit clique/cover/flow-cover cuts, `NodefileStart` disk offload, `MIPGap`, seeded RNG. `time_limit` (default 60s) applies uniformly across backends. No warm-start logic in any wrapper.

**Fallback behavior**: if the OR-Tools GUROBI backend returns an empty-result sentinel (heuristic signal of a failed solver creation), the dispatcher silently falls back to the native Gurobi path.

**Limitations/scalability caveats**: a monolithic arc-based MILP (O(n²) binary/continuous variables) with no decomposition — unlike BPC it has no column generation, so it scales only as far as the chosen MIP solver's B&B reaches within `time_limit`; `max_dist = 6,000,000 km` is an arbitrary "always feasible" arc filter, not real pruning. Since three backends reimplement the same constraints independently, there is no single source of truth for the formulation.

## HGS (Hybrid Genetic Search)

**Location & engines.** `logic/src/policies/route_construction/meta_heuristics/hybrid_genetic_search/`. `HGSPolicy._run_solver` (`policy_hgs.py`) projects lat/lng to a local equirectangular plane (centred on the depot) before calling `dispatcher.run_hgs` — required because SWAP*'s polar-sector pruning uses `atan2`, and raw lat/lng would distort angles by ~22% at mid-latitudes. `dispatcher.run_hgs` branches on `params.engine`: `"pyvrp"` delegates to `pyvrp_wrapper.solve_pyvrp` (builds a genuine `pyvrp.ProblemData`, single depot, one `VehicleType`, calls `pyvrp.solve(...)` — a pure black-box call with no VRPP profit objective; profit/skip logic is faked post-hoc). The **default engine is `"custom"`**, meaning WSmart-Route's own from-scratch HGS (`hgs.py::HGSSolver`) is what actually runs in practice — not a thin PyVRP wrapper for the primary path.

**Algorithm loop** (`HGSSolver.solve`, Vidal et al. 2022 Algorithm 1, VRPP-adapted): dual subpopulations (feasible/infeasible), `4·μ` random initial giant tours, each split and locally searched. Main loop runs until `time_limit` or `n_iterations_no_improvement` (default 20000): binary-tournament parent selection (biased fitness), crossover (`crossover_rate`=1.0 default), local-search "education", evaluation, insertion into the respective subpopulation, periodic survivor selection once a subpopulation reaches `μ+λ` (μ=25, λ=`n_offspring`=40 default). Penalty capacity is adapted every 100 iterations — a quadrant heuristic unique to WSmart-Route, not in Vidal (2022): for VRPP mode it tracks rolling average node-coverage (target 40-70%) vs. average per-route margin (target 10%) over the last 100 offspring and nudges the capacity-violation penalty across four quadrants. Restart logic re-initializes the population if stuck (only when `time_limit > 0`).

**Crossover.** `route_profit_gpx_crossover` — "RP-GPX" (Route-based Profit-aware Generalized Partition Crossover), operating on decoded routes: extracts physical edges from both parents' routes, isolates uncommon-edge components, inherits whichever parent's component segments, greedily packs the remaining node pool (profit-aware), then enforces mandatory-node presence. Plain OX is referenced as a fallback in comments, but RP-GPX is what the solver actually calls.

**Local search / education.** Delegates to a shared `LocalSearch` base implementing a granular neighborhood (top-20 nearest neighbors per node), applying intra-route moves (relocate, 2-opt intra) and inter-route moves (relocate, 2-opt, swap, SWAP with polar-sector pruning — Vidal 2022's headline contribution). Optional moves (3-opt, cross-exchange, lambda-interchange, ejection chains) default off. After local search, the giant tour is reconstructed by re-interleaving unvisited nodes back into their original relative slots, preserving genetic material for skipped (unprofitable) nodes.

**Diversity management.** Biased fitness = `rank_profit + (1 - nb_elite/pop_size)·rank_diversity`, with diversity measured via broken-pairs distance (undirected edge-set Jaccard-style metric to 5 nearest neighbors) — Vidal 2022's parameterless diversity weighting. Survivor selection removes exact clones first, then the worst-fitness individual, re-ranked incrementally via a cached pairwise-distance matrix with O(P) inverse-index eviction rather than O(P²) per removal — an engineering optimization beyond the paper.

**Split algorithm (`split.py::LinearSplit`).** A dynamic program decoding the giant tour into routes in effectively linear time via a monotonic deque (Prins 2004 / Vidal 2022 Split DP, shortest-path over an auxiliary DAG where each feasible sub-segment is an edge weighted by its route cost). Extended for VRPP with a **skip transition** that lets the DP bypass any non-mandatory node whose inclusion would reduce total profit — this is literally how HGS decides the profitable node subset, at the Split-decoding layer rather than the genetic-operator layer. Both unlimited-fleet and fixed-fleet variants exist, with a greedy fallback if the DP fails.

**VRPP subset handling.** The giant tour is genotype (all nodes); `routes` is phenotype (only profitable/visited nodes) — unvisited nodes form an "implicit dummy route" whose order only evolves under crossover pressure. Evaluation pre-filters the giant tour with a break-even hurdle (`revenue >= round-trip cost`) before Split, unless the node is mandatory.

**Limitations/citations.** No true parallelism despite a `PARALLELIZABLE` policy tag (single-threaded). Cites Vidal, T. (2022), Hybrid genetic search for the CVRP: Open-source implementation and SWAP\ neighborhood*, Computers & Operations Research 140, 105643, and Prins (2004) for Split.

## ALNS (Adaptive Large Neighborhood Search)

**Location & engines.** `dispatcher.py::run_alns` branches on engine: `"package"` wraps the real PyPI `alns` library (minimal — 2 destroy + 1 repair operator, clearly a demo path); `"ortools"`; **default `"custom"`** → `alns.py::ALNSSolver`, an 810-line from-scratch Ropke & Pisinger (2005/2006) implementation that is what actually runs.

**Main loop.** Initial solution via greedy construction, then up to `max_iterations` (default 5000) or `time_limit` (default 60s). Each iteration: roulette-wheel operator selection (`p_j = w_j/Σw_i`), `n_remove` sized per Ropke & Pisinger §4.3.1 bounds (`lower_bound=min(n,4)`, `upper_bound=min(100, 0.4·n)`), destroy then repair, profit computed. Acceptance via injected criterion (default Boltzmann-Metropolis / classic SA, geometric cooling `alpha=0.995`); start temperature dynamically calibrated (`t_start = (start_temp_control·best_profit)/ln(2)`, accepting a 5%-worse solution with 0.5 probability, per Ropke & Pisinger §3.5). Scoring follows §3.4: `sigma_1=33` new global best, `sigma_2=9` accepted-better-and-previously-unvisited, `sigma_3=13` accepted-worse-but-unvisited, deduped via canonical solution hashing. Weights update every `segment_size=100` iterations: `w_{i,j+1} = w_{i,j}(1-r) + r·(π_i/θ_i)`, reaction factor `r=0.1`.

**Destroy operators wired**: core set (always present) = `RandomRemoval`, `WorstRemoval`/`WorstProfitRemoval` (randomized-worst per §3.4.1), `ShawRemoval`/`ShawProfitRemoval` (relatedness-based). If `extended_operators=True`: `StringRemoval`, `ClusterRemoval`, `NeighborRemoval` (and their `*Profit` siblings) — 3→6 operators total. `profit_aware_operators` swaps every destroy op for its profit-marginal variant (targets `waste_i·R - detour_cost_i·C` instead of raw distance). Several other destroy operators exist in the codebase (`historical.py`, `sector.py`, `route.py`, `branch_and_bound.py`, `guided.py`, `multi_period.py`) but are **not wired into `ALNSSolver`** — reserved for sibling policies.

**Repair operators wired**: controlled by `regret_pool` ("regret2" | "regret234" default | "regretAll"): `Greedy`/`GreedyProfit`, `Regret2`/`Regret2Profit`, plus (regret234+) `Regret3`, `Regret4` and profit variants, plus (regretAll) `Savings`/`SavingsProfit`. Every noise-capable operator is expanded into `<Name>Clean`/`<Name>Noisy` adaptive-weight slots (paper Configuration 15) — default `regret234` non-profit-aware gives 8 repair slots. Noise is `U[-η·max_dist, η·max_dist]`, `η=0.025`, added only when a noisy slot is drawn — letting the adaptive mechanism itself learn whether noise helps, rather than a naive 50% coin-flip.

**VRPP profitable-subset handling.** Destroy operators pull nodes into an `unassigned` pool; profit-aware repair operators compute per-insertion marginal profit `waste·R - delta_dist·C` and **skip nodes with negative marginal profit unless mandatory** — this is the direct mechanism by which ALNS decides bin inclusion/exclusion; repair may leave nodes unassigned when no profitable position exists, rather than being forced to restore full coverage. `params.vrpp=True` (default) enables `expand_pool`, letting repair also opportunistically insert any currently-unvisited node, not just just-removed ones.

**Limitations/scalability.** `max_removal_cap=100` bounds destroy-segment size regardless of instance size; visited-solution hash set grows unbounded over a run; no restart/diversification mechanism analogous to HGS's population reset — a single trajectory with SA acceptance is the only escape from local optima.

**Citations.** Ropke, S., & Pisinger, D. (2006), An adaptive large neighborhood search heuristic for the pickup and delivery problem with time windows, Transportation Science 40(4), 455-472, with repeated precise section-number references (§3.3, §3.4.1, §3.4.3-4, §3.5, §4.3.1, Eq. 20, "Configuration 15").

## SANS (Simulated Annealing Neighborhood Search)

Implemented as two distinct engines behind one policy, dispatched by `SANSParams.engine` ("new" default, or "og"). Cites the primary source: **Jorge, D., Antunes, A. P., Ramos, T. R. P., & Barbosa-Póvoa, A. P., "A hybrid metaheuristic for smart waste collection problems with workload concerns", Computers & Operations Research 137 (2022) 105518** — a look-ahead heuristic + SA/neighborhood-search hybrid validated against a Portuguese waste operator (≥45% profit gain vs. current practice).

**"new" engine — `heuristics/sans.py:improved_simulated_annealing`.** Single-route (typically `max_vehicles=1`) main loop: outer cooling loop `while T_min < T`, inner `for _ in range(iterations_per_T)` (default 5000). Geometric cooling `T = T·alpha` (default `alpha=0.95`, `T_init=75.0`, `T_min=0.01`). Each inner iteration picks uniformly at random from 14 "route_ops" (`2opt, move, swap, remove, insert, move_n_random, move_n_consec, swap_n_random, swap_n_consec, remove_n_bins, remove_n_bins_consec, relocate, cross, or-opt`) plus 4 "add_ops" active only when a `removed_bins` pool is non-empty — this is how SANS folds the VRPP profitable-subset decision directly into the neighborhood: bins can be dropped into a global `removed_bins` set (profit-driven exclusion) and reinserted later. Acceptance is textbook Metropolis (`delta>0` accept; else `p=exp(delta/T)`, guarded against overflow). Stagnation handling: if no improvement for 500 iterations, temperature reheats to `T_init` (a reheating mechanism not in vanilla Kirkpatrick SA). Stops on `T < T_min` or `time_limit` (default 60s).

**Move taxonomy.** `_handle_2opt` (intra-route reversal); `move`/`move_between_routes` (inter-route relocation); `swap`/`mutate_route_by_swapping_bins` (intra-route pairwise swap); `relocate_within_route`/`insert_bin_in_route` (intra-route single relocation/best-position insertion); `cross_exchange` (inter-route segment swap); `or_opt_move` (short-chain relocation); `move_n_route_random/consec` and `swap_n_route_random/consec` (multi-bin n∈[2,5] relocations, random or contiguous); `remove_n_bins`/`add_n_bins` (removed-bin pool management at multi-bin granularity).

**"og" (LAC) engine.** A different SA loop tied to `search/random_search.py:local_search`, which is the actual selection-strategy layer: draws one of 19 named procedures via near-uniform weights (several "Add route" variants pinned to probability 0 — disabled dead code). `select/random.py` samples uniformly; `select/consecutive.py` extracts contiguous segments (modeling string/segment relocation); `select/greedy.py` is a first-improvement hill-climber used in refinement, not inside the SA loop. Temperature follows a **different, non-geometric schedule**: `T = T_initial / (i^T_param)` (power-law, indexed by iteration count). After SA, `refinement/route_search.py:find_solutions` chains: uncrossing → SA → `refinement/refinement.py:refine_solution` (invoking `search/deterministic.py` — nearest-50-candidate first/best-improvement sweep in forward order — and `search/reversed.py`, the same evaluator applied in reversed route/bin order, a simple diversification trick) → `refinement/rebalancing.py` (iterative greedy insert/remove of the removed-bins pool — the second mechanism handling profitable-subset selection).

**Limitations.** The "new" engine's operator selection is uniform regardless of solution state (no adaptive/weighted selection as in ALNS); `improved_simulated_annealing` is invoked single-route per call even though multi-route neighborhoods exist; `local_search_2`/`local_search_reversed` use full deep-copy-based rollback per evaluated candidate — expensive at scale; several procedure probabilities in the "og" engine are hardcoded to 0.

## PG-CLNS (Pheromone-Guided Cooperative Large Neighborhood Search)

PG-CLNS is explicitly a **WSmart-Route-original hybrid** — no bibliography PDF names "PG-CLNS," and no docstring cites an external paper for the combined algorithm; module docstrings only describe the design ("Combines ACO for construction and global guidance with LNS for local improvement, within a population-based framework guided by pheromones"). Components individually map to literature: the LNS layer cites **Pisinger, D., & Ropke, S. (2007), "A general heuristic for vehicle routing problems," Computers & Operations Research 34(8), 2403-2435**; the pheromone/candidate-list design (k-sparse edge pheromone, top-k neighbors per node) matches `bibliography/policies/K-Sparse Ant Colony Optimization.pdf` (Hale, 2021). No paper is cited for the destroy operators; the closest conceptual precedents are Shaw removal and swap-star/or-opt route-move literature — presented here as precedent, not origin.

**Top-level loop.** Population-based: build `population_size` (default 10) initial solutions via ant-colony construction (`max_iterations` pinned to 1 inside PG-CLNS so it acts purely as a constructive generator; `local_search=False` since "LNS handles local search"). Each outer iteration (default 50): (1) **Coaching** — every population member independently refined by a self-contained destroy/repair/SA-acceptance LNS loop (its own 100-iteration "coaching session"); (2) **Global competition** — track best-profit solution with deterministic tie-break; (3) **Pheromone update** — evaporate all, then deposit `elitist_weight/cost` on the current best-so-far solution's edges only (elitist ACS-style global update); (4) **Replacement** — worst 20% of the population regenerated by re-running construction under the updated pheromone field.

**What the pheromone actually guides.** A k-sparse (k=10) **edge**-pheromone matrix — deposits on directed arcs (i,j), not bin-inclusion decisions. Local ACS update during construction (`tau=(1-rho)·tau+rho·tau_0`, discouraging other ants from reusing the same edge within a construction batch) plus the global elitist update above. This is unambiguously **ant-colony-style solution construction** feeding pheromone-biased edge selection (pseudo-random-proportional rule, exploit-with-probability `q0=0.9` else roulette over `tau^alpha·eta^beta`) — **not** used to weight which destroy/repair/route operator LNS picks. Operator selection inside the LNS "coaching" phase is a separate, orthogonal mechanism: EMA operator weights (`lambda_decay=0.8`) via roulette wheel — structurally an ALNS reaction-factor scheme, disconnected from the pheromone trail. PG-CLNS layers ACO (global, cross-population, edge-level guidance shaping where ants explore) on top of ALNS (per-solution, operator-level adaptive weighting), rather than pheromones biasing LNS operator choice directly.

**Destroy/repair/intensification inventory and a real gap.** A full suite is exposed: destroy — random, worst, cluster, Shaw (relatedness-based), string (SISR-style contiguous-segment) removal; repair — greedy, regret-2/k, greedy-with-blinks insertion. **However, the LNS solver only wires three destroy ops (random, worst, cluster) and two repair ops (greedy, regret-2) into its actual registries** — Shaw removal, string removal, regret-k insertion, and blink-insertion are fully implemented and exported but dead code from PG-CLNS's runtime path. Intensification moves (2-opt intra, 2-opt, 3-opt intra, swap, or-opt) are likewise implemented and wired into a local-search module, but that module is only invoked when `local_search=True` in the ACO solver — and PG-CLNS's config explicitly sets that to `False`. **The LNS solver never calls local search at all**, so in the actual PG-CLNS composition, none of that intensification runs; "coaching" is destroy+repair+SA-acceptance only. This is a legitimate, code-verifiable gap between documented intent and the wired execution path.

**Profitable-subset (VRPP) handling.** Twofold: (1) construction filters candidates — a non-mandatory node is only "feasible" if revenue exceeds marginal insertion cost, and construction halts once no unvisited node is profitable; (2) LNS initialization applies the same screen, and acceptance operates on profit deltas via SA-style Metropolis, exactly like SANS's inner loop but with its own independent temperature schedule (`start_temp=100.0`, `cooling_rate=0.95`).

**Scalability notes.** Pheromone structure intentionally sparse (k-per-node heap eviction) to bound memory; candidate-list precomputation is O(n²), done once; population loop re-runs full 100-iteration LNS coaching sessions for every member every outer iteration — up to 50,000 LNS iterations per call before the 60s time limit typically dominates.

## PSOMA (Particle Swarm Optimization Memetic Algorithm)

**Source**: Liu, Wang, Jin & Huang (2006), "An Effective PSO-Based Memetic Algorithm for TSP" (LNCS 4222/ICIC 2006), cited verbatim in code docstrings. A fairly direct, well-attributed port to the profit-maximizing VRPP.

**Discrete encoding (the PSO-for-routing trick)**: the paper's **Ranked-Order-Value (ROV) rule** — a random-key / smallest-position-value (SPV) encoding. Each particle carries a continuous position vector `X ∈ ℝⁿ` (bounded `[0,4]`). `argsort(X)` yields a permutation of client IDs (sorting continuous values = ranking = tour order). The resulting giant tour is decoded into vehicle routes via the same linear-time `LinearSplit.split()` DP used by HGS (Vidal/Prins machinery) — this is also where VRPP profitability is handled.

**Velocity/position update**: exactly the paper's eq. (2)-(3): `V = omega*V + c1*r1*(pbest_X - X) + c2*r2*(gbest_X - X)`, clipped to `[v_min, v_max]`, then `X = clip(X + V, x_min, x_max)`, with `r1,r2` independent U(0,1) draws per-dimension. Defaults: `pop_size=20`, `omega=1.0` (note: the module docstring claims "ω≈0.4 forces intensive local exploitation" but the dataclass default is actually 1.0 — a doc/default mismatch), `c1=c2=2.0`, `max_iterations=200`, stagnation cutoff L=30 (gbest unchanged for 30 consecutive iterations → early stop), `time_limit=60s`.

**Memetic component**: a **Simulated-Annealing local search with adaptive operator selection** (the paper's "meta-Lamarckian learning strategy"), applied to the global-best solution once per outer iteration (not per-particle, not elite-only). The main loop alternates: (1) a full PSO sweep over the whole swarm, each particle re-decoded and pbest/gbest updated; (2) one call running SA search on gbest. A one-time training phase evaluates three neighborhood operators — swap, insert, inverse (2-opt-like segment reversal) — each for `n(n-1)` Metropolis steps to seed reward estimates; thereafter operator choice uses reward-proportional roulette with an ε=0.05 exploration floor, EMA-updated rewards (γ=0.5). SA acceptance defaults to Boltzmann-Metropolis (`T0=3.0`, cooling `λ=0.9`); moves are pre-filtered via an O(1) surrogate delta on a `biased_dist_matrix` before paying for the true O(n²) Split evaluation — a "lazy decoding" optimization not in the original paper.

**VRPP profitable-subset handling.** No separate "include bin i" variable — the giant tour always contains all clients, and profitability filtering happens entirely inside the shared `LinearSplit` DP (same skip-transition mechanism as HGS). The solver additionally precomputes a `biased_dist_matrix` where inter-node distances are discounted by each node's profit-to-cost ratio (scaled by a tuning constant), pulling profitable node-pairs "closer" so cheap surrogate SA moves are biased toward retaining high-value bins without paying for full Split evaluation each time.

**Limitations.** SA training costs `n(n-1)` Metropolis steps per operator (3 operators) just for initialization — O(n²) before the main loop starts; swarm size defaults to only 20 particles; `seed=42` default makes runs reproducible but could mask instability across seeds if not explicitly varied.

## ACO-HH (Ant Colony Optimization Hyper-Heuristic)

**Source**: Chen, Kendall & Vanden Berghe (2007), "An Ant Based Hyper-heuristic for the Travelling Tournament Problem", cited by name, with an explicit "Adaptations from TTP to VRPP" section documenting deliberate deviations.

**Confirmation it is a true hyper-heuristic, not classical ACO**: the module docstring is explicit — *"ants construct sequences of local search operators rather than node sequences. The pheromone matrix encodes transition probabilities between operators."* Confirmed in code: the pheromone matrix `tau` has shape `(n_operators+1, n_operators)` — an **operator × operator** transition matrix, not a city × city matrix (row `n_operators` is a "virtual start" row). A separate node-level sparse pheromone structure exists but is explicitly decoupled, used only by external callers, and plays no role in the HH's own operator selection.

**Low-level heuristics available** (11 total): `2opt_intra`, `3opt_intra`, `2opt_star`, `swap`, `swap_star` (Top-3 insertion cache, Vidal 2022), `relocate`, `perturb` (random k-node perturbation), `kick` (destroy-ratio kick, with a profit-biased variant), `shaw_removal` (+ profit variant), `string_removal`, `random_removal`. A separate **bootstrap portfolio** of 11 construction/insertion heuristics is used only once at construction time: greedy, greedy-with-blinks, nearest, farthest, savings, regret-2/3/4, deep, GENI, and branch-and-bound (LDS) insertion — each with a profit-aware sibling.

**Selection formula**: pure Ant-System proportional roulette wheel — `numerator = tau[current_op]^alpha * eta[current_op]^beta`, normalized to probabilities, sampled categorically. Defaults `alpha=1.0`, `beta=2.0`. Each ant builds a sequence of length `n_operators` (=11) per journey, departing from the ant's persistent current position across iterations (ants placed uniformly on real operator vertices at init, position carries over between outer iterations).

**Pheromone deposit/evaporation**: pure-AS evaporation `tau *= (1-rho)` (default `rho=0.5`), applied once per outer iteration before deposits. Deposit: for every ant whose journey improved the objective, `delta_tau = Q + journey_improvement/len(sequence)` (`Q=1.0` floor constant preventing weak-but-consistent signals from being erased by evaporation) is added to every traversed operator-transition edge.

**Heuristic-information (visibility) update**: `eta[i][j]` updated per-edge after each operator application using `lambda/((execution_time+1e-3)*edge_count)`, where `lambda` is either the paper-faithful fixed base (1.0001) or, by default in this repo, a scale-invariant `exp(improvement/Z)` with `Z` an EMA of `|improvement|` — an explicit VRPP-motivated deviation, since the paper's fixed base is not scale-invariant across differently-sized instances. Visibility decays each outer iteration.

**Difference from a "regular" meta-heuristic**: classical ACO lays pheromone on city-graph edges and directly constructs solutions; ACO-HH's ants never touch node-to-node choices directly — pheromone/heuristic-information matrices live entirely in **heuristic-space** (an 11×11 operator graph), and an ant's "journey" is a sequence of moves to apply to a base solution, not the solution itself. This matches slide 9's framing precisely: it searches the space of algorithmic move-sequences rather than the space of solutions.

**VRPP profitable-subset handling.** Two mechanisms: (1) `profit_aware_operators=True` switches every bootstrap/removal/insertion heuristic to its profit-maximizing sibling, weighing revenue-per-kg against marginal cost; (2) **Strategic Oscillation** (from the TTP paper's §III.B) — a capacity-violation penalty added to the objective halves after 10 iterations without improvement (relaxing capacity to explore infeasible-but-promising regions) and resets on improvement, indirectly letting the search reconsider which bins to include under relaxed constraints.

**Limitations.** Sequence length fixed at 11 regardless of problem size; default `n_ants=10`, `max_iterations=50`, `time_limit=30s` — comparatively shallow budgets vs. PSOMA's 200 iterations/60s; elitism syncing all ants to gbest on improvement risks premature convergence, acknowledged in the docstring as a deliberate fidelity-vs-diversity trade-off.

# Route Improvers

## CLS (Classical Local Search)

Registered as `"local_search"`/`"classical_local_search"`/`"cls"`. The main branch instantiates a `LocalSearchManager` driven by a **fixed, hard-coded operator list**, explicitly ordered "from simplest (cheapest) to most complex (expensive)": relocate → swap → 2-opt intra → or-opt(2) → or-opt(3) → 2-opt (inter-route) → swap → 3-opt intra → 4-opt intra.

The outer loop is **not textbook steepest descent** despite the class docstring saying so: each operator returns `True` on the first improving candidate it finds (confirmed in `move_relocate`'s single-candidate delta-and-apply logic — no comparison against other candidates), then the whole scan restarts from operator 1. This is a **first-improvement Variable-Neighborhood-Descent (VND)**: neighborhoods cycle in increasing complexity order, and any accepted move re-triggers the cheapest neighborhood first, rather than exhaustively evaluating every neighborhood and picking the single best move across all of them. Stopping is a local optimum with respect to all nine operators, or an iteration cap (default 500) as a safety valve.

Cost accounting is **pure travel-distance minimization gated by capacity feasibility**: `delta` is exclusively edge-distance differences; revenue (`R`) is passed into the manager but never enters the relocate/swap/2-opt delta computations — waste totals only gate feasibility (route load must not exceed vehicle capacity `Q`). **This refines the deck's framing**: CLS optimizes distance (km) only, subject to not violating capacity (kg feasibility, not kg optimization); it does not maximize profit or reshuffle waste allocation for revenue. For 100-350 node instances, CLS operates on already-partitioned per-vehicle sub-routes, so `N` per operator call is bounded by typical route size, not fleet-wide node count, keeping the O(N³)/O(N⁴) 3-opt/4-opt operators tractable in practice.

## Fast-TSP

Registered as `"fast_tsp"`/`"ftsp"`/`"tsp"`. Splits the tour into per-vehicle sub-routes, and for every trip with more than one node calls the third-party **`fast_tsp` PyPI package** (C++ core with pybind11 bindings) after rescaling the sub-distance-matrix to integers (the package requires an integer, symmetric, triangle-inequality-respecting `uint16` matrix).

The package's own documentation states: "Run a local solver to find a near-optimal TSP tour. For small problems, the exact solution is returned." Its exact solver is Held-Karp DP, `O(2ⁿ·n²)`, with the package's own guidance: "not recommended for n > 20" — this is the concrete origin of the deck's "~20 nodes" crossover claim (inferred from the package's documented guidance, since the code never calls the exact/heuristic sub-functions directly — it always calls `find_tour`, which decides internally). Beyond that regime, the package's own docs label the fallback explicitly as **"An SLS (Stochastic Local Search) solver for TSP"** — the exact move-set inside the closed C++ binary is not verifiable from source in this repo, so should be described as "SLS, internals not verified" rather than guessed at (e.g. don't claim it's specifically 2-opt or specifically simulated annealing without hedging).

Because Fast-TSP operates strictly on `(distance matrix, node subset)` per already-partitioned route, it **only reorders visitation sequence within a fixed route/vehicle assignment** — it cannot move bins between vehicles, drop bins, or reason about revenue/waste; it purely minimizes intra-route distance, the same axis as CLS's core moves but via one global per-route reordering rather than incremental swaps.

# Core Search Operators (2-opt, Or-opt, Swap/Relocate, Node Exchange)

**2-opt** (a `k=2` special case of k-opt): removes two non-adjacent edges `(u,next_u)` and `(v,next_v)` and reconnects by reversing the segment between them — edges become `(u,v)` and `(next_u,next_v)`. Per Shen Lin (1965), this eliminates a "crossing" iff the new edge sum is smaller; a single reversal costs O(segment length), and a full first-improvement sweep is O(N²) per pass. "Steepest" 2-opt would evaluate all O(N²) pairs and apply the single best delta; WSmart-Route's implementation is first-improvement (returns on the first accepted pair).

**Or-opt** generalizes relocate to chains of length 1-3: removes a contiguous chain (severing 2 edges) and reinserts it elsewhere (breaking 1 edge, adding 2), evaluated against all target positions. Per the Or-opt literature (Wu, Du & Song 2024), it is a standard iterated-local-search component; O(N²) per chain length per pass, first-improvement here.

**Swap / Relocate**: Relocate ("shift") removes a single node from between its neighbors and reinserts it after another node, delta computed in O(1) per candidate pair (O(N²) full sweep). Swap ("interchange") exchanges the positions of two nodes without removing either from the tour — also O(1) per pair / O(N²) sweep. Both accepted only via a cost-scaled epsilon threshold, i.e. pure distance-cost minimization; capacity feasibility gates inter-route moves but the objective carries no revenue/profit term.

**Node exchange between routes** is realized via swap* (inter-route swap with best-insertion repositioning, not just fixed-slot exchange) and a standalone `node_exchange_steepest.py` improver — a separate registrable policy, not composed inside CLS's fixed operator list. Worth flagging if pressed: CLS's internal node-exchange (swap, swap*) is first-improvement, while the standalone "steepest" node-exchange module genuinely implements best-of-neighborhood selection — a real behavioral distinction between two options visible in the same factory.

**K-opt generally**: for k>2, exhaustive move evaluation grows as O(N^k), matching the O(N³)/O(N⁴) costs of 3-opt/4-opt intra — the reason CLS orders operators cheapest-first and only escalates after cheaper neighborhoods are exhausted.

# Mandatory Selection Strategies — Additional Depth

*(Look-Ahead is already covered on-slide in full mathematical/algorithmic detail — see the deck's slide 7. This section adds depth on the other two strategies and compares all three.)*

## Last-Minute (LM)

`selection_last_minute.py`. A pure, memoryless, per-bin greedy threshold check: `fill_ratio = current_fill/max_fill`, then `mandatory_mask = (fill_ratio >= threshold)`. There is genuinely **zero lookahead** — no accumulation rate, no variance, no horizon; each bin is judged in complete isolation against a scalar fill percentage. "CF" in the docstrings/configs stands for **"Critical Fill"** level, not "confidence factor" (that term is reused for SL). The two deck variants are literally `last_minute_cf70` (`threshold: 70`) and `last_minute_cf90` (`threshold: 90`) in config. An optional EOQ-derived per-bin threshold path exists (Economic-Order-Quantity `Q* = sqrt(2·D·S/h)` batch-size formula) but is not used by the CF70/CF90 configs — those use the flat scalar threshold.

## Service-Level (SL)

`selection_service_level.py`. Core formula is a linear worst-case projection over a horizon: `predicted_fill = current_fill + accumulation_rate·horizon_days + threshold·std_deviation·horizon_days`, flagging bins mandatory when `predicted_fill >= 100`. Structurally a safety-stock-style bound `μ_horizon + z·σ_horizon`, but the confidence value is populated from config key `confidence_factor`, not a literal "z_alpha" derivation — the config docstring explicitly calls it a rough Gaussian quantile proxy (`confidence_factor=0.84 ≈ 80th percentile`), so it is used as a z-score but is a hand-set constant, not computed via `scipy.stats.norm.ppf`. Both `μ` (accumulation rate) and `σ` (std deviation) come from the same online Welford estimator described under Simulator Behavior below — not a separately known/assumed sigma, but an empirically-tracked running statistic that starts at zero and only becomes informative after several simulated days (cold-start bias for early-horizon days). **SL1 vs SL2 differ purely in `horizon_days`**: SL1 uses `horizon_days=1` (next-day projection), SL2 uses `horizon_days=2`, both at `confidence_factor=0.84` — so the deck's "SL1/SL2" split is a horizon-length ablation, not a confidence-level ablation.

## Comparative Notes (LA vs LM vs SL)

**Complexity**: LM is strictly O(n) per day — one vectorized comparison, no simulation. SL is also O(n) per day — a closed-form linear projection, no iteration — asymptotically as cheap as LM, though it requires maintaining O(n) running mean/variance state across days. LA is the expensive one: it performs actual forward simulation/rollout of future days per candidate bundling decision, scaling with the lookahead horizon and branching considered — orders of magnitude more expensive per decision than LM/SL.

**Theoretical properties**: LM offers **no probabilistic guarantee whatsoever** — a deterministic reactive rule with no model of future accumulation; overflow risk is purely a function of how conservative the fixed threshold is (CF70 vs CF90 trade risk vs. visit frequency by hand-tuning, not by any statistical target). SL is explicitly constructed to approximate a **service-level / probabilistic overflow bound**: under a (locally) Gaussian assumption on daily accumulation, `μ+z·σ` over the horizon caps the probability that true fill exceeds capacity at roughly a `Φ(z)`-controlled confidence — but this is only as good as (a) the Gaussian approximation of what's actually Gamma/empirical-generated waste, and (b) the quality of the online-estimated μ,σ (cold-start bias, no autocorrelation correction). LA is best understood as a heuristic/greedy approximation to an optimal multi-day bundling/DP policy — no closed-form guarantee, but it directly optimizes forward-simulated outcomes rather than relying on a distributional assumption, trading computation for adaptivity.

**What else exists but isn't in this deck**: all three register through the same `MandatorySelectionFactory` machinery as ~25 other strategies present in the codebase but not benchmarked here — e.g. `selection_cvar.py` (CVaR risk-tail selection), `selection_wasserstein.py` (distributionally-robust selection with a Wasserstein ambiguity radius), `selection_rollout.py` (Monte-Carlo rollout), `selection_whittle.py` (Whittle-index restless-bandit selection), `selection_learned.py` (imitation-learned classifier), `selection_dispatcher_thompson.py`/`selection_dispatcher_portfolio.py` (bandit strategy-selection meta-policies), plus knapsack/Lagrangian/submodular-greedy/savings variants. A much larger experimental surface than the three baselines this talk focuses on — a natural "what else we tried / what's available for future work" answer if pressed on why only LA/LM/SL were benchmarked.

# Simulator Behavior & Properties

**Daily state transition**: orchestrated by `run_day()` (`logic/src/pipeline/simulations/day_context.py`) via a fixed Command-Pattern pipeline: Fill → Mandatory Selection → Route Construction → Route Improvement → Collect → Log. The bin-fill physics: waste lost to overflow is computed before clamping — `todays_lost = max(real_fill + todays_filling - 100, 0)/100 * volume * density` (kg) — then fill is clamped to 100%. Overflow is flagged when `real_fill == 100.0` exactly — a **boolean saturation flag**, not a graded penalty; the graded cost is carried separately as `kg_lost`. Reset-on-visit zeroes both true and observed fill for collected bins and increments the collection counter; profit is `sum(collected)*revenue - cost*expenses`. **Unvisited overflowing bins do not carry the lost mass forward** — the excess is discarded, and fill stays capped at 100%, so a bin left unvisited keeps silently losing waste every subsequent day. Sensor noise (Gaussian) is applied to produce the observed fill used by policies, distinct from ground-truth fill used for the true overflow/loss accounting — a realistic observability gap between what a selection strategy "sees" and what actually happens. Daily reward is `total_collected - new_overflows - cost` (note: overflow count, not `kg_lost`, is the penalty term in reward, while profit is tracked as a separate primary objective).

**Capacity Q / feasibility**: not enforced centrally by the simulator loop — the simulator hands vehicle capacity and the mandatory-bin set to whichever route-constructor adapter is configured, and trusts that adapter's own capacity logic (hard MILP constraint for BPC/exact solvers, penalty or repair-based for metaheuristics). The simulator only recomputes route cost afterward; final km/profit are recomputed during the Collect step. Default `VEHICLE_CAPACITY = 200.0 kg` for synthetic instances, passed through config rather than hardcoded per environment.

**Waste-generation stochastics**: two generator families. **Gamma-k**: draws `rng.gamma(alpha, theta)` per bin from preset `(α,θ)` pairs tiled heterogeneously across bins (e.g. "gamma3" resolves to alternating blocks of `α=1` with θ alternating 8/6, and `α=3` for other bin groups) — a genuinely heterogeneous, fitted 3-parameter-family Gamma model, not one global Gamma. **Empirical**: literal bootstrap resampling of historical daily-fill records from a dataset or grid, not a parametric fit. Per-bin accumulation-rate statistics used by selection strategies (means, std) are **not** taken from the generator's true parameters — they are estimated online during simulation via **Welford's algorithm**, i.e. adaptive running estimates from realized fills (unless a precomputed stats CSV is preloaded).

**Vehicle/depot assumptions**: `n_vehicles=1` by default and not overridden in any policy config used in this benchmark — confirming the deck's stated limitation of a single vehicle/depot per region. Heterogeneous-fleet machinery is not visibly wired for actual multi-vehicle dispatch elsewhere either. An implicit max-route-duration proxy exists via a hop-count cap sized to instance scale (√n heuristic) for some environment variants; the core waste-collection environments have no explicit time-window term.

**Fairness/reproducibility mechanism**: RNG seeding is policy-isolated — a `policy_seed` is derived from `(seed + hash(canonical_policy_name) + day)` and reseeds all RNG streams per (policy, day, sample), ensuring every algorithm sees **identical bin-fill draws** for a fair comparison — this is the concrete mechanism behind the deck's "identical demand realisations" fairness claim on slide 4. Warm-starting across days is handled by carrying a multi-day context object forward between `run_day` calls rather than any environment reset.

# Additional Result Conclusions

*(Computed directly from `public/global/simulation/simulation_summary.csv` / `simulation_summary_90d.csv` as a Q&A backup — these refine, and in one case revise, the existing "most robust constructor" framing on the conclusions slide, so read them as a supplement, not a replacement.)*

**Pareto-front membership rate (30-day horizon, per scenario)**: recomputing which constructor appears on the non-dominated (min overflows, max kg/km) front in each of the 6 scenario×distribution combinations gives ACO-HH and PG-CLNS tied at the top (4/6 scenarios, 67%), followed by BPC and HGS (3/6, 50%), PSOMA (2/6, 33%), ALNS and SANS (1/6, 17%), and SWC-TCF (0/6). This slightly refines the deck's existing "BPC and PG-CLNS are the most frequent Pareto-front constructors" bullet — the more precise statement is that **ACO-HH ties PG-CLNS for the most frequent front membership**, with BPC a close second alongside HGS.

**Execution time spread is large and worth having ready**: mean per-run wall-clock time at 30 days ranges from ACO-HH (~620s) and PSOMA (~1090s) at the fast end, through SWC-TCF/ALNS/BPC (~1100-1200s), up to PG-CLNS (~2040s), HGS (~2440s), and SANS (~2590s) at the slow end — roughly a 4x spread between the fastest and slowest constructor at the same horizon. At 90 days the spread widens further and reorders somewhat (ACO-HH ~1680s fastest; SANS ~9760s mean, ~25000s max slowest) — the extra-heavyweight metaheuristics (SANS, HGS, PG-CLNS) get proportionally more expensive as the horizon grows, while ACO-HH stays comparatively cheap throughout. This is the concrete backup for the "simulation run execution time" KPI and the natural answer to "what did it cost to run BPC/exact methods vs. the heuristics" — note BPC is not the most expensive constructor by wall-clock; several metaheuristics cost more time per run than the exact B&B-and-price solver at this problem scale.

**Horizon robustness is more nuanced than "stable 30→90 days"**: mean overflow counts blow up sharply for some constructors between the 30-day and 90-day horizons — SWC-TCF (~40x), HGS (~5.4x), BPC (~3.6x), PG-CLNS (~2.7x) — while ACO-HH, PSOMA, and SANS stay close to flat (~1.0-1.1x). Median overflow ratios (less sensitive to outliers) tell a similar but softer story: SWC-TCF ~6.9x, HGS ~4.6x, BPC ~4.0x, PG-CLNS ~2.75x, vs. ACO-HH ~1.75x, SANS ~1.44x, PSOMA ~1.12x. **Caveat for Q&A**: the mean-based numbers are heavily influenced by a small number of extreme outlier rows (three 30-day rows and two 90-day rows exceed 500 overflows, all concentrated in the Figueira da Foz / Gamma-3 / 350-bin scenario, mostly under the LA strategy paired with SWC-TCF — e.g. a single 90-day SWC-TCF/LA/FFZ-350/Gamma-3 cell reaches 23,886 overflows). This reads as a specific pathological configuration (a large, high-variance Gamma-3 instance where SWC-TCF's monolithic MILP formulation likely times out or degrades badly under LA's bundling behavior) rather than a universal property of SWC-TCF — if asked "is SWC-TCF just bad at scale," the fairer answer is "one demanding scenario dominates its mean; its median degradation (~7x) is still real but less extreme," and this is exactly the kind of result that motivates the future-work item on richer, constructor-specific timeout/fallback handling at long horizons. Either way, **ACO-HH's low sensitivity to horizon extension across both mean and median is the most defensible "most robust" claim** to make confidently under questioning, refining the deck's existing "ACO-HH is the most robust across scenarios and horizons" bullet with concrete numbers rather than leaving it qualitative.

**kg/km efficiency is comparatively stable across horizons for every constructor** (deltas of roughly ±0.3-1.1 kg/km on a scale of ~5-8 kg/km baseline, both directions), supporting the deck's existing "KG/KM stays consistent" claim without needing revision — efficiency does not degrade the way overflow counts do, consistent with efficiency being driven mainly by route geometry (which doesn't change much with horizon length) rather than by cumulative overflow risk (which mechanically compounds the longer bins go unvisited).

# Limitations & Future Work — Additional Remarks

Beyond the deck's stated limitations (single vehicle type/depot per region, waste models calibrated on limited data, 90-day runs restricted to 30-day Pareto-front policies), the codebase research above surfaces several more specific, code-verifiable caveats worth having ready:

- **Exact methods are not exact under the hood in this benchmark's time budget.** Both BPC (60s default time limit, `max_bb_nodes=1000`, RCSPP label caps) and SWC-TCF (60s default MIP time limit) are configured with tight wall-clock budgets for this benchmark's scale, and both have documented non-exact fallback paths on timeout (BPC falls back to CG warm-start greedy routes; SWC-TCF's OR-Tools wrapper falls back to the native Gurobi path on a solver-creation failure signal). Reported "exact method" results should be understood as "best incumbent found within budget," not necessarily certified optimal, for larger instances (FFZ-350 in particular) — this is a defensible, code-grounded answer if challenged on whether BPC/SWC-TCF results are truly optimal.
- **PG-CLNS's documented operator suite is broader than what actually runs.** Shaw removal, string removal, regret-k insertion, blink-insertion, and all five intensification moves (2-opt intra/star, 3-opt intra, swap*, or-opt) are implemented and exported by the codebase but not wired into the LNS solver PG-CLNS actually uses — a concrete "known gap between design intent and current wiring" that is honest to disclose rather than risk being caught out on, and a natural, low-effort future-work item (wiring the already-implemented operators back in) rather than new algorithm development.
- **CLS optimizes distance only, not profit.** As detailed above, CLS's neighborhood moves compute deltas purely from edge-distance differences; profit/revenue never enters the acceptance criterion, only capacity feasibility does. A profit-aware local search variant (mirroring the profit-aware destroy/repair operators already used elsewhere in ALNS/PG-CLNS) is a natural, concrete extension distinct from the deck's existing "neural constructors" future-work bullet.
- **Fast-TSP's large-instance behavior is a third-party black box.** The SLS fallback beyond ~20 nodes runs inside a closed compiled binary; this repo cannot fully verify its internal move set from source. If asked about its worst-case behavior or theoretical guarantees beyond the exact-solver regime, the honest answer is "delegated to a well-used third-party package documented as stochastic local search; internals not independently verified here."
- **SL's confidence parameter is a hand-set constant, not a computed quantile**, and both its mean/variance inputs are cold-start-biased online (Welford) estimates rather than ground-truth distribution parameters — so SL's implied overflow-probability guarantee is only as good as the Gaussian approximation and the estimator's convergence, particularly early in a simulation horizon. This nuances any claim that SL provides a formal statistical guarantee.
- **A much larger experimental surface already exists in the codebase but isn't benchmarked here**: roughly 25 additional mandatory-selection strategies (CVaR, Wasserstein-DRO, Monte-Carlo rollout, Whittle-index bandit, imitation-learned, bandit-based meta-selection among strategies, etc.) beyond LA/LM/SL, and a comparable breadth of additional route constructors beyond the eight benchmarked (matheuristics, learning-heuristic hybrids, additional hyper-heuristics). This directly supports and extends the deck's "neural constructors in the same pipeline" future-work bullet — the pipeline's plugin architecture (uniform factory/registry pattern across all policy families) is designed to make swapping in any of these, including neural ones, a configuration change rather than an integration project.
- **Horizon-dependent instability (see Additional Result Conclusions above) suggests a concrete methodological future-work item**: rather than only extending the improver/constructor set, adding horizon-adaptive time budgets or constructor-specific fallback/repair strategies for long-horizon runs (especially for SWC-TCF and HGS, whose overflow counts grow fastest with horizon length) would directly address an observed weakness rather than a hypothetical one.
